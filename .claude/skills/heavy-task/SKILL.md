# Heavy Task — A級タスクの全権遂行ループ

正本: `AI_HEAVY_TASK_PLAYBOOK.md`（仕分け基準・テンプレ・実測ログ）。このスキルはその実行手順。

## Step 1: 仕分け（B/C級なら引き受けない）

プレイブック§1で判定する。**B級以下なら上位の型は使わず**、該当エージェント（frontend/copywriter/ops等）への依頼を提案して終了。A級の条件＝全工程が必要／参照資料3つ以上／途中判断2回以上。

## Step 2: 発注書を確定する（往復は1回まで）

次の3点が揃っているか確認。欠けていても質問攻めにしない——**自分の解釈で3点を埋めた発注書ドラフトを先に提示し、「違う箇所だけ直してください」と確認は1つだけ**にする（CLAUDE.mdの「質問は1回の応答に1つまで・まず自分の解釈で答える」に従う）：

```
■ 背景（何のためか・1〜2行）
■ 完成の定義（何がどんな形式で揃ったら終わりか。「QA通過」を必ず含める）
■ 裁量の範囲（勝手に進めてよい範囲／確認が必要な範囲／正本ファイルの指定）
```

## Step 3: 全権実行（ここが本体）

1. **材料を全部読んでから書く**。記憶・推測で数値や状態を書かない。ドキュメントと実体（コード・git）が食い違ったら実体が正
2. **決めて、理由を書く**。「案を並べて丸投げ」しない。判断に迷ったら保守側を選び、理由を1行残す
3. **未決事項はプレースホルダ**（【締切日】等）で先へ進める。プレースホルダの一覧と確定手順を必ず成果物に含める
4. **計算はスクリプトで検証**する。手計算・暗算の表を成果物に書かない
5. 実装系なら `python3 build_index.py` 等のビルド・テストを実行して確認する

## Step 4: QAループ（完成の定義に含まれる）

1. `hacos-qa` エージェントに成果物をレビューさせる（文書系の観点：事実整合・日付曜日・実績表現・リンク・内部整合。実装系はhacos-qa標準の6観点＝表示崩れ・アクセシビリティも含める）
2. **高/中の指摘がゼロになるまで修正**（最大3周。3周で収束しなければ残課題を明記してユーザーに報告）
3. 低の指摘も安ければ直す

## Step 5: 資産化と報告

1. 成果物をコミットして指定ブランチにpush（mainへは触らない・ユーザー承認後のみ）
2. `AI_HEAVY_TASK_PLAYBOOK.md` の実測ログに1行追記（ユーザーに使用量%の記録を促す）
3. ユーザーのフィードバックから好み・修正傾向が読み取れたら、プレイブック§7に追記
4. 報告は**結論先出し・正直に**。失敗した工程・飛ばした工程はそのまま言う

## 禁止事項

- 実績数値・出典の捏造（プロジェクト共通ルール）
- 1往復で終わる作業への適用（価値が出ない。実測: B級コピー改善はOpus 35 vs Fable 29）
- ユーザー確認なしの不可逆操作（main反映・外部送信・削除）
