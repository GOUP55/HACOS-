## (a) 申込〜開催の運営自動化設計

### 全体フロー（標準モデル・第2期以降を想定）

```
申込 → スタッフ通知（既存の「新規予約」通知パターンを流用）
     → 入金案内を送信（当日中）
     → 申込から3日以内に入金確認（現金またはSTORES）→ 入金で席確定
        └ 3日経過して未入金 → 参加者へ再確認＋スタッフへアラート
     → 開催7日前 → 最終確認（アレルギー・食事量・朝の時間枠）
     → 開催5日前 → 食事変更・アレルギー申告の最終締切（以降、食事部分は原則返金不可）
     → 開催（Day0〜Day7）
```

### ⚠️ 第1期には標準モデルがそのまま当てはまらない（先に共有する前提）

第1期は **申込締切7/31（金）→ Day0が8/1（土）** で、標準モデルの「開催7日前」「開催5日前」を単純計算すると次のようになる。

| 基準日（8/1開催） | 日付 |
|---|---|
| 開催7日前 | 7/25（土） |
| 開催5日前 | **7/27（月）** |
| 申込締切 | 7/31（金） |

**「開催5日前」の7/27は、申込締切7/31より前**。つまり7/28〜31に申込む人には、標準の「5日前食事締切」が原理的に成立しない（`WELLNESS_JOURNEY_MASTER_SPEC.md` リスク#25で既に指摘されている問題と同じ）。

第1期はマスター仕様書リスク#25の**案A（7/29から、その時点で入金済みの人数分の見込み製造を開始）をClaude推奨案としているが未確定**（GOと食事チームの選択待ち）。本パックの(b)テンプレは、この前提を踏まえ日付を固定値で書かず「◯月◯日」のプレースホルダにしている。第1期の実運用では、7/29前後を目安にスタッフが個別に日付を確定して送る運用を推奨する。

**盲点**: この日付の詰まりは自動化だけでは解決しない（見込み製造の判断は人が下す）。実装を待たず、まず(d)の手動チェックリストで第1期を回すのが安全。

### 実装案（既存LINE Harnessの前日リマインドパターンを拡張）

現状の`line-reservation/src/reservation-routes.js`は、**毎日JST18時に走るcron（`sendReminders(env)`）**が、D1の`sessions`/`reservations`テーブルを日付一致で検索してLINE配信する構成（前日リマインド・月末の回数券再購入リマインドで実績あり）。同じパターンにJourneyの申込データを乗せる設計が現実的。

**新規テーブル案**（`journey_applications`。マイグレーションファイル名・作成はHarnessセッションの担当）

```sql
CREATE TABLE journey_applications (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  line_user_id TEXT NOT NULL,
  name TEXT,
  cohort TEXT DEFAULT '1',
  applied_at TEXT,              -- JST日付 'YYYY-MM-DD'
  payment_status TEXT DEFAULT 'unpaid',  -- unpaid / paid / cancelled
  payment_confirmed_at TEXT,
  discount_type TEXT,           -- none / monitor / futari
  day0_date TEXT,                -- 開催週が確定したら設定 例 '2026-08-01'
  notes TEXT
);
```

**cronの拡張イメージ**（既存`sendReminders(env)`に相乗り。新しいcron定義は不要）

```js
// 疑似コード。実装はHarnessセッションで既存関数に追記する想定
async function sendJourneyReminders(env) {
  const todayJst = ...; // 既存コードのnowJst2と同じ算出方法を流用

  // 1) 入金リマインド：申込から3日経過・未入金
  const { results: unpaid } = await env.DB.prepare(`
    SELECT * FROM journey_applications
    WHERE payment_status = 'unpaid' AND date(applied_at, '+3 days') = ?
  `).bind(todayJst).all();
  for (const row of unpaid) {
    await pushToUser(row.line_user_id, [{ type: 'text', text: PAYMENT_REMINDER_TEXT }], env);
    await pushToUser(STAFF_ID, [{ type: 'text', text: `⚠️ ${row.name}さん 入金未確認（申込${row.applied_at}）` }], env);
  }

  // 2) 開催7日前確認・3) 開催5日前食事締切も同様に
  // day0_date から逆算した日付一致（date(day0_date, '-7 days') / '-5 days'）で判定し、
  // payment_status='paid' の人だけに送る
}
```

**重要な注意（自動化の範囲）**: 「入金確認」そのもの（現金の受領・STORES決済の照合）は自動化しない。`payment_status`を`'paid'`に切り替える操作はスタッフの手動更新のままにするのが安全（誤確定で席確定を早めてしまうリスクを避ける）。自動化するのは「未入金の見落とし防止アラート」と「確定済みの人への案内送信」の2点。

**実装難易度・優先度（推定）**
| 項目 | 難易度 | 優先度 | 根拠 |
|---|---|---|---|
| テーブル追加＋cron拡張 | 中（既存パターンの延長。半日〜1日程度と推定） | 高 | 入金確認の抜け＝席確定の抜けに直結。定員5名なので1件のミスの影響が大きい |
| 入金ステータスの手動更新UI | 中〜高（現状はDB直操作かシートになる） | 中 | 当面は(d)の手動チェックリストで代替可能 |

第1期の募集開始までの時間（残り14日）を考えると、**第1期はこの自動化を待たず(d)の手動運用で回し、実装は第2期に向けて並行して進める**のが現実的。
