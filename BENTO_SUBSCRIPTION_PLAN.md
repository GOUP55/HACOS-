## ◆ プラン設計案（2プラン）

### 前提となる現行単価

- `hacos-hmc-lp.html` 記載の現行弁当単価：**¥1,300〜1,500/食**（前日までLINE注文・テイクアウト可）
- 以下の試算は単発注文を続けた場合の想定額と、定期便価格を比較した「お得感」の例。**実際の価格はオーナーが原価・仕込みコストを見て最終決定してください。**

### 週3食プラン（仮）

| 項目 | 内容 |
|------|------|
| 食数 | 週3食 × 月4週 ＝ 月12食 |
| 単発換算 | ¥15,600〜18,000（@¥1,300〜1,500） |
| 定期便価格（仮） | **¥14,800/月** |
| お得率 | 単発下限比 約5.1%OFF／単発上限比 約17.8%OFF（幅があるため「約5〜8%お得」目安として提示） |

### 週5食プラン（仮）

| 項目 | 内容 |
|------|------|
| 食数 | 週5食 × 月4週 ＝ 月20食 |
| 単発換算 | ¥26,000〜30,000（@¥1,300〜1,500） |
| 定期便価格（仮） | **¥24,500/月** |
| お得率 | 単発下限比 約5.8%OFF／単発上限比 約18.3%OFF（同上の目安） |

**設計メモ：**
- 単発単価に¥1,300〜1,500の幅があるため、定期便は「メニューによらない定額」にすることで会計・発注をシンプルにする（栄養価・ボリュームは週替わりでも価格は固定）。
- 「単発比5〜10%お得」を謳うなら、定期便価格は単発**下限（¥1,300）**基準で5〜10%引いた額に寄せるのが安全（上限基準で計算すると割引率を誇大に見せてしまうため）。上記の仮価格は下限基準でおよそ5〜6%引きに設定した例。オーナーが原価と相談の上、最終的な割引幅を決めてください。

## ◆ 注文フロー：LINEでの「ゆるいサブスク」運用案

現状確認：LINE Harness（`line-reservation/` 配下、`line-harness.hacos.workers.dev`）は現時点で**予約・回数券機能**が実装済み（`DEPLOY_KAISUKEN.md` / `LINE_HARNESS_HANDOFF.md` 参照）。**週次の「注文確認→スキップ可」を自動送信する機能は未確認（＝現状は無い前提で設計）。** 自動化するなら別途開発が必要。

### 運用ステップ（MVP：自動化なしでも回せる形）

```
① 毎週◯曜日（例：木曜18時）
   GOがLINE公式の「一斉配信」機能で定期便利用者に確認メッセージを送信
   → 「今週も通常通りお届けします。お休みする場合はこのトークに
      『今週スキップ』と送ってください」

② 締め切り（例：翌日金曜21時）までに返信が無ければ「通常通り」とみなす
   → GOまたはスタッフが返信一覧を見て仕込み数を確定

③ 「今週スキップ」の返信があった人はその週だけ注文数から除外
   → 請求（月額制の場合）はスキップ回数分を翌月に繰越 or 日割り調整
      （繰越／日割りのどちらにするかはオーナー決定事項に追加検討）

④ 受け渡し日（HMC開催の日曜 or 指定曜日）にお渡し
```

### 将来の自動化案（開発が必要・今回は設計のみ）

- LINE Harnessに「毎週◯曜◯時に対象タグの友だちへ一斉送信 → キーワード返信で回数券的にスキップ扱い」を実装できれば、GOの手動配信作業がゼロになる。
- 実装難易度：中（回数券機能の応用で作れる可能性あるが、要スケジューラ・返信パース処理の追加開発）
- 優先度：低〜中（まずは手動運用で需要を検証してからでよい）

### 解約自由の明記（心理障壁を下げる設計）

告知文面・LP・LINEメッセージすべてに以下を必ず入れる：

> 「いつでもスキップ・解約OK。縛りはありません。」

理由：サブスク導入時の最大の離脱要因は「縛られる不安」。解約自由を先に明言することで初回申込のハードルを下げる（根拠：一般的なサブスクCROの定石。本プロジェクト固有の実績データではない点に留意）。

## ◆ 損益の考え方：食数が読めることのメリット

| 項目 | 都度注文（現状） | 定期便（週3食/週5食プラン） |
|------|----------------|---------------------------|
| 発注タイミング | 前日確定（読みにくい） | 週次で確定食数が事前に分かる |
| 仕入れロス | 見込み違いで廃棄・不足リスクあり | 定数仕入れで廃棄ロス減・欠品リスク減 |
| 仕込み効率 | 毎日個別対応（バッチ化しにくい） | 週の総量が分かるためバッチ調理・まとめ買いがしやすい |
| 原価率 | 少量仕入れで原価率が高くなりがち | まとめ仕入れで原価率改善の余地（具体的な改善率は仕入れ交渉次第・捏造数値は出さない） |
| キャッシュフロー | 都度払い | 月初一括請求にすれば前受金化できる（決定事項#4次第） |

**ポイント：** 割引額（5〜10%）の原資は「値引きのための値引き」ではなく、上記の仕入れ・仕込み効率化で吸収する設計にする。原価が改善しない場合は割引率を見直す前提で運用開始することを推奨。
